## 第一阶段：健康检测

### 检测项目

| 检测项 | 检测方法 | 预期状态 | 超时 |
|--------|----------|----------|------|
| API 主站 | GET / → 200 | `{"ok":true}` | 5s |
| API 数据库连接 | D1 query test | success | 10s |
| 网站可访问 | GET blastjunior.com → 200 | HTTP 200 | 10s |
| 关键 API 端点 | GET /teams, /news, /members, /chat | `{"ok":true}` | 5s each |
| 磁盘空间 | df -h | >10% available | 2s |
| Worker 响应时间 | 请求耗时 | <500ms | 10s |

### 健康状态定义

```
GREEN  - 所有检测通过
YELLOW - 部分检测异常，但不影响核心功能
RED    - 核心功能不可用，需立即处理
```

---

## 第二阶段：问题分类

检测到问题后，自动分类：

| 问题类型 | 定义 | 处理策略 |
|----------|------|----------|
| `api_timeout` | API 超时 | 重试 3 次 exponential backoff |
| `db_error` | 数据库错误 | 检查连接、重试、记录 |
| `deploy_failed` | 部署失败 | 自动回滚到上一个版本 |
| `data_stale` | 数据过期 | 通知管理员更新 |
| `perf_degraded` | 性能下降 | 记录指标、通知 |
| `content_flagged` | 内容违规 | 加入审核队列 |

## 第三阶段：自动修复

### 自动修复规则

```
问题: api_timeout
操作: 
  1. 重试 (delay=2s)
  2. 重试 (delay=4s)  
  3. 重试 (delay=8s)
  4. 如果还是失败 → 标记 RED → 通知管理员

问题: db_error
操作:
  1. 检查 D1 连接状态
  2. 执行简单 query 测试
  3. 如果失败 → 检查 Cloudflare status
  4. 记录错误详情

问题: data_stale
操作:
  1. 检查最后更新时间
  2. 发送通知给管理员
  3. 不自动修改（需要人工确认）
```

### 自动修复能力范围

**可以自动修复 ✅：**
- API 超时重试
- 简单的连接问题
- 性能指标记录
- 日志清理

**不能自动修复，需要通知 ⚠️：**
- 代码 bug
- 数据库结构问题
- 需要老K提供新数据
- 重大配置变更

## 第四阶段：知识库

### 记录内容

```json
{
  "incident_id": "uuid",
  "timestamp": "2026-04-18T19:00:00Z",
  "problem_type": "api_timeout",
  "detected_at": "...",
  "resolved_at": "...",
  "resolution": "auto_retry_success",
  "attempts": 2,
  "knowledge_gained": "当 API 超时时，等待 4 秒重试成功率最高",
  "outcomes": "success"
}
```

### 自我学习规则

```
1. 如果某个修复策略成功 3 次 → 加入"已验证方案"
2. 如果某个修复策略失败 2 次 → 标记为"低成功率"，尝试其他方案
3. 如果新问题没有匹配方案 → 记录为"新问题"，通知管理员
```

---

## 第五阶段：进化引擎

### 学习方向

| 维度 | 学习内容 | 进化方式 |
|------|----------|----------|
| **性能** | API 响应时间趋势 | 记录基线，异常时报警 |
| **可用性** | 系统 uptime | 自动计算 |
| **内容** | 用户互动数据 | 分析热门内容 |
| **错误** | 错误模式 | 归类并更新修复规则 |

### 知识演进

```
第1次遇到问题 → 记录 → 等待管理员处理
第2次遇到问题 → 复用上次方案
第3次遇到问题 → 方案加入知识库
第N次遇到问题 → 直接应用已验证方案
```
