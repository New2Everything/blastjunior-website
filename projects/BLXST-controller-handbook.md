# BLXST Controller 工作手册

> **Description**: Controller 通用工作原则
> **维护规则**: 不要增加骨架

> 通用工作原则，不预设固定模板

---

## ⚠️ Spawn 五要点（每次必须遵循）

| 要点 | 说明 |
|------|------|
| **输入** | 任务描述 + 参考文档 |
| **执行** | 子智能体的具体行动 |
| **输出** | 交付物（文档/代码/报告） |
| **工具** | 用什么技能/工具完成 |
| **知识** | 基于什么业务知识设计 |

```
每次 spawn 前必须列出这 5 点，否则任务无效
```

## 🔄 设计合规性检查（新增）

在设计阶段和开发阶段之间，必须检查：

| 检查项 | 说明 |
|--------|------|
| **Rules 遗漏** | 设计是否包含 API 约束、数据来源等 rules 规定的要点 |
| **完整性** | 是否有遗漏的功能/页面 |
| **技术可行性** | 设计是否可实现 |
| **测试标准** | 是否为每个页面/功能/展示位制定详细测试标准 |

### 检查流程

```
Architect 设计完成
    ↓
主控检查设计是否符合 Rules
    ↓
主控检查设计是否包含详细测试标准
    ↓
如果不符合 → 退回 Architect 修改
    ↓
如果符合 → 进入开发阶段
```

### 常见检查项

| Rules 要求 | 设计必须包含 |
|------------|--------------|
| 单 API 架构 | API 端点设计 |
| 数据来自 D1/R2/KV | 数据来源说明 |
| CORS 单域名 | 安全配置 |
| 禁止硬编码 | 数据获取方式 |
| **测试标准** | **每个页面、功能、展示位的测试方法** |

### 测试标准要求

**Architect 在设计时必须为每个设计项制定测试标准：**

| 设计项 | 必须包含的测试标准 |
|--------|-------------------|
| **每个页面** | - 页面加载正常<br>- 数据展示正确<br>- 无技术字段泄露<br>- 无JS错误 |
| **每个功能** | - 功能可用<br>- 交互正常<br>- 状态正确 |
| **每个展示位** | - 数据来自数据库<br>- 显示格式正确<br>- 无数据缺失 |

**示例格式：**

```markdown
### 页面：战队列表

| 测试项 | 测试方法 | 验收标准 |
|--------|----------|----------|
| 页面加载 | curl HTTP 200 | 页面可访问 |
| 数据展示 | agent-browser | 战队名称显示，无"club ID" |
| 数据完整性 | 对比API | API返回数量 = 页面显示数量 |
| JS错误 | agent-browser errors | 控制台无error |
```

---

### 反馈机制

如果测试脚本调不到相关规则 → 反馈给 Architect 改进设计文档

## 核心思路

每次 spawn 子智能体，主控需要思考：

```
1. 输入：这次任务需要什么信息？
2. 行动：这个子智能体应该做什么？
3. 输出：我需要它交付什么？
```

---

## 输入层

### 主控需要给子智能体的输入

| 输入 | 来源 | 每次需要思考 |
|------|------|--------------|
| 任务描述 | 老K指令 | 这次要做什么？ |
| 参考文档 | 项目知识 | 需要什么知识？ |
| 约束条件 | rules/md | 有什么限制？ |
| 测试方法 | skills | 用什么工具？ |

### 项目知识清单

主控通过以下方式知道当前项目知识：

```
1. 进入项目时执行：ls projects/*.md knowledge/*.md
2. 每次查看最新文件列表
3. 知识会随项目增加而增加
```

### 技能清单

主控通过以下方式知道可用技能：

```
1. 执行 ls skills/ 查看已安装技能
2. 使用 find-skills 发现新技能
3. 技能会随安装增加而增加
```

### 思考方式
```
不是"每次都给所有文档"
而是"这次任务需要哪些文档？"
```

---

## 行动层

### 子智能体能做什么

| 能力 | 来自 |
|------|------|
| 设计 | Architect 角色 |
| 开发 | Coder 角色 |
| 测试 | Tester 角色 |
| 搜索 | find-skills 角色 |
| ... | ... |

### 思考方式
```
不是"这个任务给 Architect"
而是"这个任务需要什么能力？"
```

---

## 输出层

### 子智能体能交付什么

| 输出 | 例子 |
|------|------|
| 设计文档 | 方案、规格 |
| 代码 | 部署完成 |
| 测试报告 | 通过/失败 |
| 分析报告 | 调研结果 |
| ... | ... |

### 思考方式
```
不是"Architect 一定输出文档"
而是"我需要什么交付物？"
```

---

## 禁止事项（通用规则）

| 禁止 | 原因 |
|------|------|
| 输入为空 | 子智能体不知道做什么 |
| 不给参考文档 | 可能违反项目规范 |
| 照搬现有实现 | 要创新设计，不是复制 |
| 不检查输出 | 可能交付不合格 |

## 主控的知识和技能

### 主控已知

| 类型 | 例子 |
|------|------|
| 项目知识 | design.md, rules.md, knowledge-graph.md, status.md |
| 项目技能 | agent-browser, website-test, wrangler, ai-news |

### 主控未知

| 情况 | 行动 |
|------|------|
| 缺少某项知识 | 先查 docs/、memory/、projects/、knowledge/，无则询问老K |
| 缺少某项技能 | 先用 find-skills，无则询问老K |
| 不知道有哪些知识 | 执行 `ls projects/*.md knowledge/*.md` 查看 |
| 不知道有哪些技能 | 执行 `ls skills/` 或用 find-skills 查看 |

### 思考方式
```
每次 spawn 前：
1. 思考：这个任务需要什么知识？
2. 思考：这个任务需要什么技能？
3. 先尝试自己解决（查文档、find-skills）
4. 如果解决不了 → 询问老K
```

## 经验教训

| 问题 | 解决 |
|------|------|
| 只检查 API 不检查 UI | Tester 必须用 agent-browser |
| Coder 没按 Design | 输入必须包含 design.md |
| 主控忘记检查 | 主控要定期抽查 |

*思考方式比固定模板更重要*
