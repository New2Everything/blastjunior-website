# 兰星少年俱乐部官方网站 设计白皮书 v1.0

## 基本信息

**定位**：无人值守 AI 运营管理官网（对外好逛好玩；对内可管理可追溯）

**风格**：明亮运动风（Bright Sports）

**用户**：成员 / 家长 / 粉丝 / 潜在新人 / 赞助合作方

---

## 0. 成功标准

- 每周自动更新：新闻 / 赛果 / 赛程 / 相册（至少一类持续更新）
- 首页形态：体育头条 + 人物 + 精彩瞬间
- 成员可用：选手/战队页（战绩、照片、分享卡、荣誉）
- 权威口径：积分榜/赛果/荣誉可追溯
- 闭环齐全：产生 → 审核 → 发布 → 分发 → 复盘（可追溯）

## 2. 功能优先级

### P0（必须）
- 首页聚合：赛果 / 倒计时 / 本周活动 / 相册视频 / 最新故事
- 赛程赛果：赛季/组别/战队筛选；比赛详情页
- 积分榜：赛季/组别；胜/负/积分/净胜；可追溯
- 战队页：简介 / 排名 / 近期战绩 / 成员 / 相册 / 荣誉
- 选手页：头像 / 昵称 / 称号 / 本赛季数据 / 里程碑 / 相册 / 徽章
- 画廊：赛事/训练/活动/人物；时间线/标签；自动封面与归档
- 加入我们：加入俱乐部 / 活动报名 / 商务合作；自动回执；进入待处理队列
- AI运营后台：草稿箱 → 自动审核 → 发布排程 → 分发 → 复盘

### P1（重要）
- 比赛复盘页：比分 / 阵容 / 亮点摘要 / 精选照片 / 关联人物与战队
- 一键分享卡：比赛 / 选手 / 战队 / 荣誉 / 活动（统一样式）
- 订阅关注：战队 / 选手 / 赛季；站内更新提示
- 专题策展：紫龙之路 / 新星计划 / 训练营 / 家长指南
- 搜索与标签：人名/队名/赛事/地点/关键词；结果可按标签缩小

### P2（拓展）
- 徽章任务：首胜 / 连胜 / 里程碑 / 训练打卡
- 现场大屏：赛程 / 对阵 / 积分 / 荣誉
- AI问答导览：问答 + 跳转链接
- 多语言：英/日优先

**视觉**：浅底留白 + 高能点缀；运动元素（斜切角/速度线/计分牌/徽章）

- **品牌主色**：导航/主按钮/链接/关键标题
- **行动点亮色**：报名/加入/分享/关注
- **对抗色**（红/蓝）：仅对阵/比分/半场
- **背景**：浅雾色；分割轻线

### 排版与照片
- **文案**：短段落；三行原则
- **照片**：自然光、动作清晰；每场至少一张封面级
- **卡片**：统一圆角与阴影

### 动效
- **卡片**：轻抬起
- **比分**：翻牌/弹入
- **徽章**：轻提示

## 14. Cloudflare Stack（端到端数据流）

### 1. 组件与边界（谁负责什么）

| 组件 | 职责 |
|------|------|
| **用户浏览器** | 访问页面、触发API请求、加载图片/视频 |
| **Cloudflare Pages** | 前端托管/CDN、静态页面与路由、静态资源（样式/脚本/字体/图标） |
| **Cloudflare Workers** | 唯一API门面、聚合/列表/详情/搜索/推荐位、鉴权与权限分层、数据聚合、缓存与限流 |
| **Cloudflare D1** | 权威结构化数据：赛季/组别/战队/选手/比赛/赛果/积分/文章/专题/标签/报名/审核记录 |
| **Cloudflare R2** | 媒体对象存储：原图/缩略图/中图/封面图/分享卡/视频 |
| **Cloudflare KV** | 热点加速与状态：首页聚合快照/热点榜单快照/推荐位快照/配置/开关 |
| **Queues/定时触发** | 异步任务：生成缩略图/分享卡/封面、夜间重算榜单、批量发布、巡检告警 |

### 2. 部署链路

- **Git仓库作为唯一发布源**
- 提交/合并触发Pages自动构建
- 自动部署/可回滚

### 3. 前端路由（稳定URL）

| 路由 | 说明 |
|------|------|
| `/` | 首页 |
| `/matches` | 赛程列表 |
| `/matches/{matchId}` | 比赛详情 |

**要求**：每个详情页稳定URL + 标题摘要封面图（用于分享预览）

### 4. 数据流（四条主链路）

#### 4.1 赛事链路（赛程/赛果/积分）
1. 数据进入（录入/导入/对接）
2. 写入D1（比赛、赛果）
3. 触发积分更新写入D1
4. Worker生成首页/榜单快照写入KV（短TTL）
5. 前端：列表页拉分页数据、详情页拉单场数据、首页拉聚合数据（优先KV）

#### 4.2 内容链路（新闻/战报/活动/人物/荣誉）
1. AI产出草稿
2. 自动审核（事实/合规/风格）
3. 发布写入D1（文章/专题/标签/关联对象）
4. 推荐位/专题索引写入KV
5. 前端按栏目/专题/标签拉取展示

#### 4.3 媒体链路（相册/封面/分享图）
1. 原始媒体进R2
2. 异步生成：缩略/中图/封面/分享卡
3. D1记录媒体元数据
4. 前端默认加载缩略/中图；原图仅在放大或下载时加载

#### 4.4 加入链路（报名/合作/咨询）
1. 用户提交表单
2. Worker接收并写入D1（报名/合作线索/状态）
3. 自动回执（站内提示 + 可扩展到社群通知）
4. 后台队列：待处理列表 + 状态流转

### 5. 关键原则

- **前端不直连D1/R2/KV**；所有数据口径由Worker统一
- **D1是权威**；KV是加速与推荐；R2是媒体；Pages是展示
- 列表必须分页；大相册必须分段加载；图片必须分层
- 热点页面优先KV快照；更新后刷新快照或短TTL自然过期
- 任何发布都可追溯：来源/依据/审核/版本

### 10. 缓存命中与降级策略

**缓存命中目标**：
- 首页聚合：≥80%
- 积分榜：≥80%
- 热点详情：≥60%

**降级策略**：
- KV不可用 → 直接查D1（功能可用优先）
- D1慢 → 返回"上一次快照 + 更新时间提示"
- R2图片慢/失败 → 先显示占位图与文字信息

### 11. 监控与告警（必须具备的观测面）

- **Worker**：响应时间、错误率、缓存命中率

- **关键页面**：首页/赛程/积分/画廊的可用性与加载耗时采样

## 15. 数据分离原则（开发规范）

### 数据类型定义

| 类型 | 定义 | 示例 | 存储 | 访问 |
|------|------|------|------|------|
| **可变化数据** | 会变化的内容 | 赛果/积分/排名/比赛记录 | R2 JSON | Worker API |
| **可统计数据** | 可被计数的内容 | 在线人数/浏览量 | R2/KV | Worker API |
| **可追溯数据** | 需要历史记录 | 积分榜/历史战绩 | R2 | Worker API |
| **品牌表达** | 固定文案 | Slogan/介绍/口号 | 静态HTML | 直接渲染 |

### 开发规则

**⚠️ 真实数据原则**：D1/R2/KV中存储的是真实世界事件，禁止伪造。如需修改数据，必须由老K告知处理方式。开发过程中需要数据时，应询问老K获取。

1. **变化数据必须经由Worker API** - 禁止前端直连D1/R2/KV
2. **品牌表达允许占位** - 与数据模块严格隔离
3. **API降级策略** - 优先读真实数据，读不到用默认数据兜底

### 当前数据流

```
前端 → Worker API → D1 (新闻/赞助商)
                → R2 JSON (战队/选手/比赛)
                → R2 (照片/媒体)
```
