# MEMORY.md - 重要经验总结

## ⚠️ 数据真实性原则（最高优先级）

**任何时候都不能捏造数据！**

根据设计白皮书，数据分为两类：

| 类型 | 定义 | 来源 | 示例 |
|------|------|------|------|
| **数据模块** | 可变化/可统计/可追溯 | **必须来自D1/R2/KV** | 赛果/积分/排名/战队/选手 |
| **品牌表达** | 固定文案/口号/介绍 | 允许占位或AI创作 | Slogan/导航/版权声明 |

**核心原则：**
- 数据模块必须来自真实数据库，经由Worker API
- 如果数据库为空，那就是空，不能用假数据填充
- 网站测试的data_test.sh就是为了验证数据是否来自真实数据库

## ⚠️ 数据模块与展示的双重机制（网站开发核心原则）

### 机制说明

开发网站时，处理数据模块展示的决策流程：

```
想要展示某个数据模块（如战队列表）
        ↓
数据库是否有这个数据？
        ↓
是 → 开发展示模块，把现有数据展示出来
否 → 询问老K：
       - 需要这个展示？→ 建数据库，后续老K提供数据
       - 不需要？→ 不开发这个展示模块
```

### 两条铁律

1. **不展示无数据之模块**
   - 如果数据库没有数据，就不应该开发对应的展示模块
   - 除非老K明确说要加这个模块

2. **不遗漏已有之数据**
   - 如果数据库里已经有数据，应该尽量在网站中展示
   - 如果有未展示的数据，必须询问老K是否需要展示

### 保底约定

- **数据库数据不能捏造** - 是就是，不是就不是
- **展示数据必须来自数据库** - 禁止硬编码数据模块内容

### 实际应用举例

| 场景 | 正确做法 |
|------|----------|
| 想加"选手排行榜"但数据库无数据 | 询问老K是否需要 |
| 数据库有4个选手，网站只显示3个 | 询问老K是否需要展示全部 |
| 数据字段是"team1"但页面用"team" | 修复字段映射，不是捏造数据 |
| 数据库为空但页面显示"暂无数据" | ✅ 正确（允许占位表达） |

### 验证方法

每次开发完成后，用 `data_test.sh` 验证：
- 数据是否来自API
- 数据库数据是否都被展示

## Cloudflare D1 权限问题

**问题**: API Token有D1权限，但直接调用API返回 "Route not found"

**原因**: D1的API路径格式与Workers/KV不同，官方文档中的路径可能不正确

**解决方案**: 使用 `wrangler` CLI操作D1，功能完全正常

```bash
CLOUDFLARE_API_TOKEN="xxx" CLOUDFLARE_ACCOUNT_ID="xxx" wrangler d1 list
```

## agent-browser 使用技巧

- 非常适合自动化测试网站
- 可以模拟用户登录、点击、填表等操作
- 截图功能强大，支持全页截图
- 适合排查UI bug

## Workers + R2 公开访问

- R2默认不能直接公开访问
- 需要创建Worker绑定R2 bucket
- Worker处理/public/*路径，返回R2文件内容
- 适用于网站静态资源托管
